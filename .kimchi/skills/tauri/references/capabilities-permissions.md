# Capabilities & Permissions (Tauri v2)

C'est le changement de sécurité majeur par rapport à v1. En v1, toutes les commandes enregistrées étaient accessibles par défaut (`allowlist` servait juste à activer/désactiver des blocs d'API). En **v2, l'accès est deny-by-default** : chaque commande (core ou plugin) doit être explicitement autorisée pour une fenêtre/webview donnée.

## Vocabulaire

- **Permission** : décrit un privilège explicite pour une ou plusieurs commandes (ex. `fs:allow-write-text-file`). Peut aussi définir un *scope* (quels chemins/URLs sont concernés).
- **Permission set** : un identifiant qui regroupe plusieurs permissions (ex. `fs:default` regroupe un ensemble raisonnable de permissions fs). Les plugins fournissent souvent un set `default`.
- **Capability** : un fichier JSON/TOML qui associe des permissions à des fenêtres/webviews (par `label`). C'est la capability qui "active" réellement les permissions pour une fenêtre donnée.
- **ACL** (Access Control List) : le mécanisme sous-jacent qui relie identifiers de permission ↔ commandes ↔ scopes.

## Où ça vit sur le disque

```
src-tauri/
├── tauri.conf.json
├── capabilities/
│   └── default.json      # capability(ies) — JSON, JSON5 ou TOML
└── permissions/          # uniquement si tu définis TES PROPRES permissions custom (rare pour une app, courant pour un plugin)
    └── <identifier>.toml
```

Tous les fichiers du dossier `capabilities/` sont chargés automatiquement (sauf s'ils sont explicitement référencés autrement dans `tauri.conf.json`).

## Exemple de capability minimal

```json
// src-tauri/capabilities/default.json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "Capability par défaut pour la fenêtre principale",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "dialog:default",
    "shell:default"
  ]
}
```

- `windows` : liste de labels de fenêtres (glob autorisé, ex. `"*"`) auxquelles s'applique la capability.
- `permissions` : liste d'identifiants — format `<plugin>:<permission>` (le préfixe `tauri-plugin-` est ajouté automatiquement, pas besoin de l'écrire). Pour des commandes définies directement dans l'app (pas un plugin), pas de préfixe : juste `<permission-name>`.
- `platforms` (optionnel) : restreint la capability à certains OS (`linux`, `windows`, `macOS`, `iOS`, `android`).
- `remote` (optionnel, à utiliser avec une extrême prudence) : autorise des origines distantes (pas seulement le bundle local) à utiliser l'IPC — pertinent seulement si le frontend charge du contenu depuis une URL distante.

## Workflow pour ajouter l'accès à une nouvelle commande de plugin

1. Trouver la permission exacte dans la doc du plugin ou dans son dossier `permissions/autogenerated/` sur le repo source (souvent nommée `allow-<command_name>` ou groupée dans `<plugin>:default`).
2. L'ajouter dans le tableau `permissions` de la capability qui couvre la fenêtre concernée.
3. Si la commande a besoin d'un scope (ex. quel dossier `fs` est accessible), ajouter un objet au lieu d'une simple string :

```json
{
  "identifier": "fs:allow-write-text-file",
  "allow": [{ "path": "$APPDATA/myapp/*" }]
}
```

4. Relancer `tauri dev` — **il n'y a pas de vérification à la compilation**, l'erreur `X not allowed` n'apparaît qu'à l'exécution dans la console webview.

## Commandes propres à l'application (pas un plugin)

Pour une commande définie dans `src-tauri/src/lib.rs` avec `#[tauri::command]`, Tauri génère automatiquement `allow-<nom_commande>` et `deny-<nom_commande>` — il suffit généralement d'ajouter `allow-<nom_commande>` à la capability, sans écrire de fichier de permission séparé, sauf besoin de scope custom.

## Capabilities différentes par fenêtre/plateforme

Utile pour une app avec plusieurs fenêtres (ex. fenêtre principale + fenêtre "settings") qui n'ont pas besoin des mêmes accès :

```json
{
  "identifier": "fs-desktop-only",
  "windows": ["main"],
  "platforms": ["linux", "windows", "macOS"],
  "permissions": ["fs:default"]
}
```

Bonne pratique : un fichier de capability par catégorie fonctionnelle plutôt qu'un seul fichier monolithique — plus facile à auditer.

## Content Security Policy (CSP)

Configuré dans `tauri.conf.json` sous `app.security.csp` :

```json
{ "app": { "security": { "csp": "default-src 'self'; script-src 'self'" } } }
```

⚠️ Mettre `"csp": null` **désactive complètement** la CSP — à éviter en production sauf raison très spécifique documentée.

## Erreurs fréquentes

| Symptôme | Cause probable |
|---|---|
| `Error: X not allowed. Permissions associated with this command: ...` | Capability manquante ou scope trop restrictif — le message liste les permissions valides à ajouter |
| Le plugin fonctionne en dev mais pas en build release | Vérifier que la capability est bien incluse dans le bundle final et que le `identifier` correspond au label de fenêtre réel (pas au titre) |
| Ça marche sur une fenêtre mais pas sur une nouvelle fenêtre créée dynamiquement | La nouvelle fenêtre a un `label` non couvert par le `windows` de la capability — ajouter le label ou un pattern glob |
| Accès fs/http qui marchait avant et plus après une mise à jour de plugin | Le plugin a peut-être renommé ou resserré ses permissions par défaut — comparer avec le changelog du plugin |

## Isolation Pattern (avancé, rarement nécessaire)

Pour des cas où le frontend charge du code potentiellement non fiable (ex. plugin système tiers, dépendance douteuse), Tauri propose le **isolation pattern** : une application JS "secure" injectée en iframe sandboxée qui intercepte tous les messages IPC avant qu'ils n'atteignent le Core, avec chiffrement via `SubtleCrypto`. À ne considérer que si l'app charge réellement du contenu non maîtrisé — pour une app comme Reqly avec un frontend 100% maîtrisé, ce n'est généralement pas nécessaire.
